Project Overview

What Is This Project?

This is a full-stack web application — meaning it has two separate programs that work together:

  1. Backend (server) — runs on your machine, handles data and AI logic
  2. Frontend (client) — runs in the user's browser, handles what people see and interact with

The app lets users browse real estate properties, chat with an AI agent named Mira to search by filters (location, budget, bedrooms), and compare properties side by side.

Theory — Full Stack

"Full stack" means the entire technology stack from the user's screen down to the server:

  • Frontend (the "front" the user sees) — HTML, CSS, JavaScript running in a browser
  • Backend (the "back" the user never sees) — a server program handling data, logic, and external APIs
  • Database (in this project: JSON files) — where the data lives

The frontend and backend are separate programs running on different ports (3000 and 5000). They communicate over HTTP.

Foundational Theory

Node.js

Node.js is a JavaScript runtime — it lets you run JavaScript outside the browser. Normally JavaScript only runs inside web browsers (Chrome, Firefox), but Node.js takes the V8 engine (Chrome's JS engine) and packages it as a standalone program.

Why It Matters

The backend is written in JavaScript. Without Node.js, JavaScript could only run in browsers — you couldn't build a server with it.

Key Properties:

  • Single-threaded: One thread handles all requests (unlike Java/C# which create a thread per request)
  • Event-driven: Uses an "event loop" — instead of blocking while waiting for I/O, it registers a callback and moves on
  • Non-blocking I/O: fetch(), file reads, database queries — none of these freeze the server

npm (Node Package Manager)

npm is the package manager for Node.js. A "package" is a reusable library someone else wrote.

  • package.json — the manifest file. Lists what packages your project needs
  • node_modules/ — the folder where npm downloads all packages
  • npm install — reads package.json and downloads everything
  • npm start — runs the start script defined in package.json

Our backend's dependencies:

The ^ means "compatible with" — ^5.2.1 means any version >=5.2.1 and <6.0.0.

What Is an API?

API = Application Programming Interface. It's a contract: "send me data in THIS format, and I'll send you back data in THAT format."

In our project, the backend exposes a REST API — the frontend sends HTTP requests to specific URLs (called "endpoints"), and the backend responds with JSON data.

Example
Frontend sends:    POST http://localhost:5000/search
                   Body: { "location": "New York", "budget": 500000 }

Backend responds:  [{ "id": 1, "title": "3 BHK Apartment...", ... },
                    { "id": 6, "title": "Cozy Studio...", ... }]

JSON Format

JSON (JavaScript Object Notation) is a text format for structured data. It's universal — every programming language can read/write it.

{
  "id": 1,
  "title": "3 BHK Apartment in Downtown",
  "price": 450000,
  "amenities": ["Gym", "Swimming Pool", "Parking"]
}

Rules: keys must be double-quoted; values can be strings, numbers, booleans, null, arrays, or objects; no trailing commas; no comments.

  • JSON.parse(text) — converts a JSON string into a JavaScript object
  • JSON.stringify(obj) — converts a JavaScript object into a JSON string

HTTP Protocol

HTTP (HyperText Transfer Protocol) is how browsers and servers communicate. Every web page load, every API call — it's all HTTP.

HTTP Request

  • Method: GET (read), POST (send data), PUT (update), DELETE (remove)
  • URL: /search, /chat, etc.
  • Headers: Metadata like Content-Type: application/json
  • Body: The actual data (for POST/PUT)

HTTP Response

  • Status Code: 200 (OK), 400 (bad request), 404 (not found), 500 (server error), 502 (bad gateway)
  • Headers: Metadata
  • Body: The response data (usually JSON for APIs)

Environment Variables

Environment variables are key-value pairs that exist in the operating system's environment. Used to store secrets (API keys) and configuration.

GROQ_API_KEY=gsk_migfHREZss2NtMlwwMop...
Why Not Hardcode?

Security: If you push code to GitHub with a key in it, anyone can steal it.

Flexibility: Different keys for dev/staging/production environments.

Separation: Code is logic; config is data. They change for different reasons.

The dotenv library reads a .env file and loads each line into process.env:

require("dotenv").config({ path: ".env" });
console.log(process.env.GROQ_API_KEY); // "gsk_migfHREZss2NtMlwwMop..."

Backend (server.js — Port 5000)

Express.js

Express is a web framework for Node.js. It wraps Node's raw HTTP module with a cleaner API for routing, middleware, and request/response helpers.

const express = require("express");
const app = express();   // creates an Express application instance

app is the core object. You attach routes and middleware to it, then tell it to listen on a port.

Middleware

Theory — What Is Middleware?

Middleware is a function that sits between the incoming request and your route handler. It can read/modify the request, read/modify the response, end the request, or pass control to the next middleware via next().

Middleware runs in order — the order you call app.use() matters.

1
Request arrives
↓
2
app.use(cors())

Adds CORS headers to the response

↓
3
app.use(express.json())

Parses JSON body → populates req.body

↓
4
Request Logger

Prints [1:03:32 AM] POST /search to console

↓
5
Route Handler

Your app.post("/search", ...) function runs

↓
6
Response sent back

express.json() — Body Parsing

When the frontend sends JSON in a POST request, the raw HTTP body is just a string. express.json() automatically parses it into a JavaScript object and puts it on req.body.

Request Logger

app.use((req, res, next) => {
  console.log(`[${new Date().toLocaleTimeString()}] ${req.method} ${req.url}`);
  next();  // MUST call next() or the request hangs forever
});

next() passes control to the next middleware/handler. If you forget it, the request never reaches your route.

CORS (Cross-Origin Resource Sharing)

Theory — The Same-Origin Policy

Browsers block JavaScript from making requests to a different "origin" (protocol + domain + port). This prevents malicious websites from calling your bank's API using your cookies.

Our frontend (localhost:3000) and backend (localhost:5000) are different origins. Without CORS, the browser would BLOCK every fetch() call.

app.use(cors()) adds these response headers, telling the browser "allow it":

Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization

Crash Handlers

process.on("uncaughtException", (err) => { ... });
process.on("unhandledRejection", (reason) => { ... });
Theory — Exceptions & Promises

An exception is an error that interrupts normal code flow. If nothing catches it (try/catch), it becomes "uncaught" and Node.js terminates the process.

A Promise represents an async operation. If it rejects and nobody handles it (.catch()), that's an "unhandled rejection." These handlers are a safety net — they log errors instead of silently crashing.

Property Data — The Merge Operation

Three JSON files are loaded and merged into one array on server startup:

FileFields
property_basics.jsonid, title, price, location
property_characteristics.jsonid, bedrooms, bathrooms, size_sqft, amenities
property_images.jsonid, image_url
 
const merged = basics.map((property) => {
  const detail = chars.find(item => item.id === property.id);
  const image  = images.find(item => item.id === property.id);
  return { ...property, ...(detail || {}), ...(image || {}) };
});
Theory — Key Operations

.map() — creates a new array by transforming each element.

.find() — returns the FIRST element matching a condition.

Spread operator (...) — unpacks an object's properties into a new object.

(detail || {}) — if detail is undefined, use empty object (prevents crash).

This is equivalent to a SQL LEFT JOIN across three tables.

Result: merged is an array of 10 property objects, each with all fields: id, title, price, location, bedrooms, bathrooms, size_sqft, amenities, image_url.

Endpoint: POST /search

POST /search
Filters properties by location, budget, and/or bedrooms.
Request:  { "location": "New York", "budget": 500000, "bedrooms": 3 }
Response: [ ...matching properties ]

The filter logic:

const result = merged.filter(home => {
  return (
    (!location || home.location?.toLowerCase().includes(location.toLowerCase()))
    &&
    (!budget || Number(home.price) <= Number(budget))
    &&
    (!bedrooms || Number(home.bedrooms) >= Number(bedrooms))
  );
});
Theory — Filter Pattern

Each condition uses the pattern (!filter || check):

  • If the filter is null/empty → !filter is true → condition passes (skip this filter)
  • If the filter has a value → evaluate the check

This is short-circuit evaluation using the OR (||) operator.

?. (Optional chaining) — if home.location is null, return undefined instead of crashing.

.includes() — checks if one string contains another: "New York, NY".includes("New York") → true.

Endpoint: POST /compare

POST /compare
Compares 2–4 properties and identifies winners per category.

Validation (defensive programming):

Highlights calculation:

const indexOfMin = arr => arr.indexOf(Math.min(...arr.filter(v => v > 0)));
const indexOfMax = arr => arr.indexOf(Math.max(...arr));
Step-by-step for "cheapest"

Prices = [450000, 380000, 850000]

1. filter(v > 0) → [450000, 380000, 850000]

2. Math.min(...) → 380000

3. indexOf(380000) → 1

4. selected[1].id → 2 (property 2 is cheapest)

What Is an LLM?

Theory — Large Language Models

A Large Language Model (LLM) is a neural network trained on massive amounts of text data. It predicts "what text should come next" given some input. Modern LLMs can understand natural language, follow instructions, and generate structured output like JSON.

What Is Groq?

Groq is an AI inference provider with custom hardware (LPUs) that runs LLM models extremely fast. The model used is Llama 3.3 70B — Meta's open-source model with 70 billion parameters. Groq provides an OpenAI-compatible API.

System Prompt — Context Injection

LLM APIs use a messages format with roles:

RolePurpose
systemSets the AI's persona, rules, and context (the "instructions")
userThe human's message
assistantThe AI's response (for multi-turn conversations)

Our system prompt injects the entire property database as context, then instructs the LLM to return a specific JSON schema. This is called in-context learning.

API Call Parameters

const groqRes = await fetch(GROQ_API_URL, {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${process.env.GROQ_API_KEY}`,
    "Content-Type": "application/json"
  },
  body: JSON.stringify({
    model: GROQ_MODEL,
    response_format: { type: "json_object" },
    temperature: 0.4,
    messages: [
      { role: "system", content: systemPrompt },
      { role: "user",   content: message }
    ]
  })
});
Theory — Key Parameters

response_format: { type: "json_object" } — forces the model to output valid JSON.

temperature: 0.4 — controls randomness. 0 = deterministic, 1 = creative. 0.4 is reliable for structured tasks.

Bearer authentication — standard HTTP auth scheme. The API checks this token to verify authorization.

await — pauses execution until the Promise resolves. Can only be used inside async functions.

fetch() — built-in function (Node.js 18+) for making HTTP requests. Returns a Promise.

Response Parsing

const groqJson     = await groqRes.json();
const responseText = groqJson.choices?.[0]?.message?.content ?? "";
responseData = JSON.parse(responseText);

Groq returns: { choices: [{ message: { content: '{"isSearch":true,...}' } }] }. We extract and parse the JSON string. If parsing fails, we fall back to using the raw text as a plain reply.

Frontend (App.js — Port 3000)

React.js

React is a JavaScript library for building user interfaces, created by Facebook (Meta). Its core idea: your UI is a function of your state.

UI = f(state)

When state changes → React re-renders → DOM updates automatically.

Theory — Key React Concepts

Components — reusable UI building blocks (functions that return JSX)

Props — data passed from parent to child (read-only)

State — data owned by a component that can change over time

Hooks — functions that let you "hook into" React features (useState, useEffect, etc.)

JSX (JavaScript XML)

JSX looks like HTML but is actually JavaScript. It gets compiled into React.createElement() calls.

<div className="card">
  <h3>{property.title}</h3>
  <p>Price: {USD.format(property.price)}</p>
</div>

Key differences from HTML:

  • className instead of class (reserved word in JavaScript)
  • {} curly braces = JavaScript expressions inside JSX
  • onClick instead of onclick (camelCase)
  • Self-closing tags required: <img /> not <img>

Virtual DOM

Theory — How React Updates the Page

React doesn't directly modify the browser's DOM. It maintains a virtual DOM — a lightweight JavaScript copy of the actual DOM tree.

When state changes:

  1. React creates a new virtual DOM tree
  2. It diffs the new tree against the old one
  3. It calculates the minimum set of changes needed
  4. It applies only those changes to the real DOM

This is called reconciliation — it's why React is fast even with frequent updates.

React Hooks

useState — State Management

const [input, setInput] = useState("");
  • useState("") creates a state variable with initial value ""
  • Returns [currentValue, setterFunction]
  • setInput("hello") updates the value AND triggers a re-render
  • Immutable: never mutate state directly — always use the setter
Why Not Mutate Directly?

React doesn't "watch" your variables. It only re-renders when you call the setter. input = "hello" changes the variable but React never knows — the UI stays stale.

Functional updates (when new state depends on old state):

setCompareIds((prev) => [...prev, id]);

Guarantees you're working with the latest value (important with rapid updates).

useEffect — Side Effects

useEffect(() => {
  fetch(`${BACKEND_URL}/properties`)
    .then(r => r.json())
    .then(data => { ... });
}, []);
Theory — Side Effects

A side effect is anything that interacts with the outside world: API calls, DOM manipulation, timers. React components should be "pure," so side effects go in useEffect.

Dependency array:

  • [] (empty) = run once after first render
  • [messages, thinking] = run when those values change
  • No array = run after EVERY render (usually a mistake)

useRef — Persistent References

const bottomRef = useRef(null);
const inputRef = useRef(null);

A ref is a mutable container that persists across renders without causing re-renders. Used for DOM references (<div ref={bottomRef} />) — bottomRef.current gives you the actual DOM element for scrollIntoView() and focus().

useCallback — Memoized Functions

const sendMessage = useCallback(async (text) => { ... }, [input, loading, allProperties, addMessage]);
Theory — Why Memoize?

Every React render creates new function instances. Passing onClick={handleClick} to a child means the child sees a "new" function each render and re-renders unnecessarily.

useCallback returns the same function instance unless the dependency values change — preventing unnecessary child re-renders.

Component: MessageContent

function MessageContent({ content }) {
  return (
    <span>
      {content.split("\n").map((line, li) => (
        <span key={li}>
          {li > 0 && <br />}
          {line.split(/\*\*(.*?)\*\*/g).map((part, i) =>
            i % 2 === 1 ? <strong key={i}>{part}</strong> : part
          )}
        </span>
      ))}
    </span>
  );
}

A mini Markdown parser that handles:

  1. Newlines: splits on \n, inserts <br />
  2. Bold: splits on **...** using regex, wraps in <strong>
Theory — Regular Expressions

/\*\*(.*?)\*\*/g means:

  • \*\* — literal ** (backslash escapes)
  • (.*?) — capture group, non-greedy (stop at first **)
  • g — global flag (find all matches)

For "I found **3 properties** in **Austin**", .split() produces: ["I found ", "3 properties", " in ", "Austin", ""] — odd indices are bold.

Component: ComparisonModal

function ComparisonModal({ data, onClose }) {
  if (!data) return null;   // render nothing when not active
  ...
}

Key mechanisms:

  • Conditional rendering: returns null when data is null (modal hidden)
  • Overlay click-to-close: clicking the dark overlay calls onClose

Theory — Event Propagation

When you click an element, the click event "bubbles" up to parent elements. Clicking the modal content would also trigger the overlay's onClick. stopPropagation() stops the event from reaching the parent.

  --bg:     #0C1628;
  --teal:   #00CEC4;
  --font-display: "Plus Jakarta Sans", sans-serif;
}
/* Used as: */ background: var(--teal);

:root is the top-level element. Variables defined here are global — change the value once, it updates everywhere.

Flexbox Layout

.layout { display: flex; }
.chat-panel { width: 380px; flex-shrink: 0; }
.results-panel { flex: 1; }  /* takes remaining space */
Theory — Flexbox

Flexbox arranges children in a row or column. flex: 1 means "grow to fill remaining space." flex-shrink: 0 means "never shrink below my set width."

CSS Grid (Property Cards)

.property-grid {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(300px, 1fr));
  gap: 20px;
}

repeat(auto-fill, minmax(300px, 1fr)) = create as many columns as fit, each at least 300px wide, stretching equally. Responsive by default — no media queries needed.

CSS Animations

@keyframes slideUp {
  from { opacity: 0; transform: translate(-50%, 20px); }
  to   { opacity: 1; transform: translate(-50%, 0); }
}
.compare-bar { animation: slideUp 0.3s ease forwards; }
Theory — Keyframes

@keyframes defines an animation sequence. slideUp moves the element from 20px below + invisible → to its resting place + visible. ease = start slow, speed up, slow down. forwards = keep the final state after the animation ends.
